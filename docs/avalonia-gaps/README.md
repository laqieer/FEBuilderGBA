# Avalonia vs WinForms — Systematic Gap-Sweep Reports

This directory contains the **canonical baselines** for the multi-axis sweep
methodology that hunts gaps between the legacy WinForms GUI and the new
Avalonia GUI. Each report is generated by a `--gap-sweep-*` CLI flag on
`FEBuilderGBA.Avalonia`. The methodology is tracked in the meta-issue
[#374](https://github.com/laqieer/FEBuilderGBA/issues/374); each phase ships
in its own PR.

## Why

The Avalonia migration was "vibe coded" — fields silently dropped, jumps
re-wired by mistake, localisation literals left as English, undo plumbing
omitted. A steady stream of issues (#355, #356, #359-365, #367-372, …) keeps
proving coverage is incomplete. Most of those bugs would be **mechanically
findable** by side-by-side comparison of the two apps. This sweep
infrastructure produces a refreshable inventory of every gap so we can fix
them as backlog instead of waiting for users to file them one at a time.

## Refresh cadence and report-churn policy

- **Baseline reports** (committed under `docs/avalonia-gaps/`) are refreshed
  **only** on phase PRs and on follow-up PRs that explicitly close one or
  more gaps and want to record the delta.
- **Advisory CI** (Phase 7, scheduled) regenerates the static-analysis reports
  (`density`, `undo`, `l10n`) on every PR and uploads them as workflow
  artifacts + job-summary previews. Those artifact runs do **not** re-commit
  the baseline.
- This split keeps the tree quiet (no daily noisy report churn) while still
  giving reviewers an instant "what changed since baseline" diff on demand.

## Generating reports locally

```powershell
# Phase 1 — density sweep against the current worktree
dotnet run --project FEBuilderGBA.Avalonia/FEBuilderGBA.Avalonia.csproj -c Release `
    -- --gap-sweep-density --out=docs/avalonia-gaps/$(Get-Date -Format yyyy-MM-dd)-density-sweep.md

# Phase 2 — field-label diff against the current worktree
dotnet run --project FEBuilderGBA.Avalonia/FEBuilderGBA.Avalonia.csproj -c Release `
    -- --gap-sweep-labels --out=docs/avalonia-gaps/$(Get-Date -Format yyyy-MM-dd)-labels-sweep.md

# Dry-run (writes only the YAML front-matter — useful for checking write perms / CI plumbing)
dotnet run --project FEBuilderGBA.Avalonia/FEBuilderGBA.Avalonia.csproj -c Release `
    -- --gap-sweep-density --dry-run --out=tmp/density-dry.md
```

All flags accept:

- `--out=<path>` — required: where the report goes.
- `--dry-run` — header-only; skips the actual scan.
- `--repo-root=<path>` — override the repo-root auto-discovery (defaults to walking up from the executable until `FEBuilderGBA.sln` is found).

## Reading the density report

The density report ranks every paired editor by `(AV controls - WF controls) / WF controls × 100 %`:

- **Negative `Δ%`** = Avalonia has **fewer** visible controls than WinForms. Strong gap signal.
- **Positive `Δ%`** = Avalonia has more (usually fine — refactor or richer UI).
- Verdicts: HIGH (|Δ%| ≥ 50), MEDIUM (25–50), LOW (<25).

The "Top-20 HIGH Gaps" subsections are placeholders for hand-written triage
notes — fill them per the suggested `grep` snippets so each subsequent
fix-PR can crib the field list directly from the baseline.

## Reading the labels report

The labels report complements the density report — density tells you WHICH
pairs probably have gaps, labels tells you WHAT specific labels appear in
the WinForms designer but are missing from the Avalonia view.

- **WF-only labels** = candidates for missing fields in the AV migration.
- **AV-only labels** = usually fine (rewording, language polish, layout change).
- **Common labels** = labels that survived normalisation on both sides.

Normalisation collapses whitespace, strips trailing colons, removes mnemonic
markers (`&` for WinForms, `_` for Avalonia), and lowercases — so `Name:` /
`&Name` / `_Name` / `Name` all collide to the same set key. The report
preserves the original casing/punctuation; only the diff key is normalised.

Per-pair sections are sorted by WF-only count descending so the biggest
candidate-missing-field counts surface first. Top of the first labels baseline
([2026-05-22](2026-05-22-labels-sweep.md)): `EmulatorMemoryForm` (174 WF-only),
`MapSettingFE7UForm` (90), `SkillConfigFE8NSkillForm` (84), `EventCondForm` (81),
`MapSettingForm` (78).
