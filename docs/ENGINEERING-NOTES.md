## Engineering Notes & Gotchas

This section collects hard-won engineering lessons, environment quirks, and recurring failure modes encountered while developing FEBuilderGBA. Each note records a concrete gotcha and its remedy. Memory claims are point-in-time; verify file:line citations against current code before relying on them.

### Avalonia GUI

- **Fluent `NumericUpDown` has a ~120px minimum width.** Setting `Width="64"` (or anything < ~120) is clamped — the control still renders at ~120px and overflows its grid column, colliding with the neighbouring control. This is the root cause of the recurring "spinners overlapping / squished" bugs (e.g. round-2 #1686/#1688 on Map Exit Point, Map Change, Skill editors; #1713 on the Event Unit move-path grid in `EventUnitView.axaml`). Fix: make each NUD's *grid column* ≥ ~128px (not just the NUD `Width`, which is ignored below 120). Verify with a headless layout test that realizes a populated row and asserts `nud[i].right <= nud[i+1].left` via `TranslatePoint(..., window)` + `Bounds.Width` — eyeballing missed a 100px-column attempt that still overlapped (pitch 100 < NUD 120). Test gotchas: the editor View is itself a `Window` (`Show()` it directly, don't nest it); FE8-gated panels are `IsVisible=false` without a ROM (force true); virtualizing `ListBox` may realize 0 rows headless (override `ItemsPanel` to a plain `StackPanel`); feed the list a public stand-in object with the bound property names.

- **List-stride mismatch bug class (#1377).** An editor's left address-list (`LoadList`) built as `base + i*STRIDE` must use a STRIDE/base that matches the record the per-entry loader (`LoadEntry`) actually reads/writes — otherwise jumps land on wrong addresses, clicking a row loads garbage, and writes can corrupt unrelated data or ROM code. Canonical case: `ImageBattleAnimeViewModel.LoadList` walked `image_battle_animelist_pointer` at stride 4, but that pointer is the 32-byte ANIME-data table; the real 4-byte SP record is a floating per-class region `p32(class+52)` (FE7/8) / `+48` (FE6). Ground truth: the WinForms counterpart's `InputFormRef(BaseAddress, BlockSize)` is the authority for a table's base+stride. Watch for slot-vs-struct: a 4-byte slot holding a u32 pointer must be dereferenced (e.g. via `MapChangeCore.PlistToOffsetAddr`) before loading an N-byte struct (the MapTileAnimation bug, #1403). Audit method: map-reduce over the ~134 list-backed VMs (`grep -rln "List<AddrResult>\|SetItemsWithIcons\|SetItems("`), keep those with a `base + i*stride` builder, then adversarially deep-audit and WinForms-cross-check only flagged candidates. A 2026-06-24 run found 3 confirmed bugs + 1 false positive (MoveCostEditor's cross-table surrogate is deliberate). Consider a `ListParityHelper` assertion.

- **WinForms owner-draw grouping vs flat list (#1254).** A WinForms ListBox using `OwnerDraw(Draw, DrawMode.OwnerDrawVariable)` often holds a NON-deduped data list (one entry per occurrence); visual collapsing happens only in the custom `Draw` method via a "draw header only when the group key changes" pattern. The Avalonia `AddressListControl` is flat (one `AddrResult` = one visible row), so a 1:1 port makes the key visibly repeat (flag 0x07 shown 8x → "Flags used: 94"). Fix: dedup the merged list by the GROUP KEY the WinForms `Draw` collapses on, keeping the first occurrence (FE8U ch0: 94 → 21). Beware the inverse trap: per-inner-scope dedup is NOT the WinForms group key and made it worse.

- **AXAML / localization traps.** `Log.Error`/`Log.Notify`/`Log.Debug` in Core join `params string[]` with spaces and do NOT substitute `{0}` — use interpolated strings (Copilot flags the composite-format bug on nearly every PR). `StringFormat` is not translated. Hardcoded hex fails DarkMode E2E. Tool/aggregator VMs that only WRITE (no ROM read for display) must NOT contain the string `IDataVerifiable` *anywhere* — even in a doc-comment — because `NoOrphanVMs_ImplementIDataVerifiable` does `source.Contains("IDataVerifiable")` over `*ViewModel.cs`; phrase comments as "orphan by the data-verification contract." Conversely, any IDataVerifiable VM must be added to `FormMappings` in `AvaloniaFieldCompletenessTests.cs` or `NoOrphanVMs` fails. Avalonia 11.2.x `Grid` has no `Row/ColumnSpacing`. When removing a deferred feature, grep ALL files (Core + axaml + axaml.cs + VM) for the stale claim.

- **New Avalonia `<Window>` view trips three cross-cutting CI gates** that functional tests don't catch (learned #1434); run locally before pushing:
  1. `AvaloniaScrollViewerTests` (Core.Tests): every `<Window>` AXAML must declare `SizeToContent="..."` AND contain a `ScrollViewer`, or be added to the `ScrollViewerExempt` set (a DockPanel-root window with internal ListBox/TextBox scrolling needs the exemption — this can fail CI even when all three local test projects are green).
  2. `AutomationIdTests.All_AutomationIds_Follow_Naming_Convention` (Avalonia.Tests): every AutomationId must end in a valid leaf suffix (`_Input _Combo _Button _List _Check _Expander _TabControl _Tab _Image _Label _Control _Link _TopBar`). A TextBox preview = `_Preview_Input`, a WrapPanel = `_Control`, a status TextBlock = `_Status_Label`. <99% compliance fails.
  3. `L10nCoverageTest.AvaloniaViews_HaveJaAndZhTranslations` (Avalonia.Tests): every English AXAML `Text=` literal needs BOTH ja AND zh entries in `config/translate/{ja,zh}.txt` (format `:key\n<translation>`). Dynamic code-behind button text isn't scanned, but route it through `R._()` and translate runtime status strings anyway.
  - Local run: `dotnet test FEBuilderGBA.Core.Tests --filter AvaloniaScrollViewerTests` and `dotnet test FEBuilderGBA.Avalonia.Tests --filter "AutomationIdTests|L10nCoverageTest"`.

- **Shared event-editor host (#1585 / PR #1589).** `EventScriptEditorCore` is script-type-agnostic (drives off the instance `_es`, not `CoreState.EventScript`); `EventScriptViewModel` is already `ScriptType`-aware (Event/Procs/AI). Procs reuses the Event engine + toolbar; AI does NOT (`AIScriptViewModel` is independently wired around the fixed 16-byte grid). Critical safety trap: `EventScriptEditorCore.Serialize` appended the FE EVENT terminator when none was present, but a Procs stream needs the Procs `End` (`0000000000000000`, 8 bytes); fix adds `EventScriptType` to the engine ctor so Procs/AI append the family `{TERM}` (shortest `ScriptHas.TERM` command) or refuse with `WriteResult.NoTerminator`. `EventScript.DisAseemble` dereferences `CoreState.CommentCache` (tests must set `HeadlessEtcCache` or NRE) and synthesizes a full 4-byte UNKNOWN on a short tail (add an `addr+size > limit` bounds guard). FE8U `Default_event_script_term_code[0]` is `0x40` (multi-byte), NOT `0x0A` — assert "appended tail is all-zero" rather than hard-coding an event-term byte. A Core null-guard can invalidate a test that relied on the NRE — grep for tests that null the thing you're guarding.

### CI & Release gates

- **`master` is protected — direct push is rejected** (`GH013`, "4 of 4 required status checks expected") and branch protection sets `required_conversation_resolution=true`. A green + MERGEABLE + auto-merge-armed PR can sit `mergeStateStatus=BLOCKED` indefinitely with 0 required checks/reviews because Copilot posts inline review threads AFTER the PR opens; unresolved threads block auto-merge. Correct merge protocol: open PR → arm auto-merge → WAIT for the Copilot review → fix legit findings / reply on non-issues → RESOLVE every thread (`gh api graphql -f query='mutation{resolveReviewThread(input:{threadId:"<ID>"}){thread{isResolved}}}'`) → confirm not conversation-blocked → then exit. "Exit immediately after arming auto-merge" and thread-resolution are not in conflict; do both. Diagnose with `gh api repos/.../branches/master/protection --jq '.required_conversation_resolution'` and a `reviewThreads` GraphQL query. Pushing a fix often auto-resolves outdated threads. Several Copilot findings are genuinely real (format bugs, over-claims, O(N×M) line counts, translate-loader corruption) — fix, don't blanket-dismiss.

- **PR screenshot-URL CI gate.** The required `build` check includes a "Validate PR screenshot URLs" step (`scripts/validate-pr-screenshots.sh`) that fails fast (~28s) when the PR *body* references a feature-branch screenshot URL (`.../blob/<feature-ref>/...png` or feature-branch raw URLs) — it is PATTERN-ONLY and never resolves the URL. CHECK 1 fails if the ref is not the default branch; CHECK 2 requires feat/fix PRs to contain at least one image markdown/`<img>`. The `issue-assets` branch is ALSO rejected by this gate (it killed PR #1612) — issue-assets is only for ISSUE-comment images, which aren't validated. Correct self-contained recipe: commit the PNG to `pr-screenshots/pr<N>-<slug>.png` ON THE FEATURE BRANCH, then reference it as `https://raw.githubusercontent.com/laqieer/FEBuilderGBA/master/pr-screenshots/pr<N>-<slug>.png` — the master-pattern URL passes the pattern check, and the file lands on master at merge so the link resolves. Diagnose with `gh run view <id> --log-failed | grep -i "screenshot URLs"`.

- **Screenshot-only helper PR Review-Gate exemption (#1959).** A helper may skip only the PR cross-model board when every canonical predicate in `DEVELOPMENT-WORKFLOW.md` is independently verified: trusted same-repository author, accepted parent plan, `docs:` title, only newly added mode-`100644` valid and visually inspected blobs under `pr-screenshots/` whose paths end in lowercase `.png`, no GitHub closing keyword paired with any issue reference in the title/body or commits, and an all-checked test plan. Its PR body must contain the exact marker block:
  ```markdown
  ## Review Gate
  Review-Gate-Exemption: screenshot-only-helper
  Parent-Plan: https://github.com/laqieer/FEBuilderGBA/issues/<N>#issuecomment-<ID>
  Parent-Issue: Ref #<N>
  ```
  The marker alone never authorizes a bypass; re-check every predicate after each push. Safety screening, CI, branch freshness, all-three-channel feedback, merge confirmation, post-merge CI, and cleanup still apply. Ordinary docs PRs and screenshot PRs with any explanatory Markdown or other non-PNG change are not exempt. The default-branch image URL becomes reachable after the helper merges.
- **Docs/screenshot helper PRs must NOT pair a GitHub closing keyword with any issue reference in the title/body or any commit message.** GitHub recognizes the case-insensitive families `close/closes/closed`, `fix/fixes/fixed`, and `resolve/resolves/resolved`. A tiny `docs:` screenshot-hosting PR titled with a bare `(#1465)` (or `Closes/Fixes #N`) will auto-close the feature issue (reason COMPLETED) when the docs PR merges — even though the real feature PR is still open with unresolved threads (false closure). Detect via `gh issue view N --json stateReason,closedByPullRequestsReferences` (a `closedBy` listing a docs/non-feature PR is a false close); confirm the feature is really merged with `git cat-file -e origin/master:<new-core-file>`. Recover with `gh issue reopen N`. Reference the issue as `Ref #N` in helper PRs; only the feature PR carries the closing keyword.

- **Issue screenshot hosting** (no PR): host on the long-lived `issue-assets` branch and reference `https://raw.githubusercontent.com/laqieer/FEBuilderGBA/issue-assets/pr-screenshots/<name>.png`. Workflow: `git fetch origin issue-assets` → `git worktree add <tmp> issue-assets` → copy PNGs into `<tmp>/pr-screenshots/` → commit → push (retry loop) → `curl` each raw URL for HTTP 200 → remove the worktree. Never delete the `issue-assets` branch (it is the permanent image host) and never switch branches in the main worktree.

- **Stale-base audit trap.** Before trusting any code audit's findings (and before filing issues), confirm the local checkout is current. A 2026-06-25 audit ran 17 commits behind `origin/master` and re-reported two HIGH "data-corruption" bugs that were already fixed upstream (#1377/#1407, #1403/#1406). Always: (1) `git fetch origin` then `git rev-list --count HEAD..origin/master` (>0 ⇒ stale); (2) `git log --oneline HEAD..origin/master` + `git diff --name-only HEAD origin/master -- <project-dir>` — only files touched by missing commits can differ; (3) re-verify each touched-file finding non-destructively via `git show origin/master:<path>` (do NOT `git checkout`/pull the shared main worktree). Workflow/Agent subagents read the working tree and inherit the stale base. An audit that rediscovers known-and-fixed bugs is a *good* accuracy signal — just dedup against what shipped.

- **Multi-issue dev-flow orchestration.** Driving N issues to MERGED via one worktree-isolated agent each: bake into agent prompts "do ALL work yourself in ONE worktree; NEVER spawn sub-agents; NEVER arm a Monitor and stop; poll CI inline with a sleep loop (`while ! gh pr checks <pr> | green; do sleep 15; done`); commit+push the moment it compiles." Agents that spawn a sub-agent or arm a background Monitor then stop ping-pong without finishing — the completion notification fires whenever an agent stops with no live children, so take over and merge rather than resume. `gh pr merge <pr> --merge --auto` is the landing lever (arm once threads are resolved). Recurring per-PR gates every agent hits: screenshot-URL (master-pattern only), L10n ja+zh, the `Log.Error {0}` composite-format bug, and `git add -A` accidentally staging multi-hundred-MB `tmp/shots` dumps (tell agents to `git add` specific files + `rm -rf tmp/shots`). Verify "Closes" vs "Ref" before closing an umbrella issue.

- **A "silent" frozen agent may have already committed complete work.** A dev-flow agent whose branch SHA sits frozen with no PR for ~2h may not be dead — it can be stuck in a screenshot/side-quest loop *after* committing a complete, tested implementation. Before relaunching from scratch: `git -C <worktree> log --oneline origin/master..HEAD` + `git status --short` + `git show --stat <sha>` to see what's actually committed; if the feature is pushed, open the PR yourself (`gh pr create --head <branch>`), `TaskStop` the stuck agent, remove its worktree, and relaunch a thin finisher scoped only to the remaining gates (explicitly telling it the feature is done). A real crash says "API Error: Connection closed"; a side-quest loop has a frozen SHA but an active task and last-result text about screenshots/polish.

- **GitHub wiki is a separate repo** (`https://github.com/laqieer/FEBuilderGBA.wiki.git`), edited and pushed directly to `master` — no PR, no CI, no branch protection. Wiki pages: `Home`, `Getting-Started`, `Editors`, `Command-Line-Interface`, `Decomp-Project-Mode`, `Avalonia-GUI`, `Native-Android`, `Skill-Systems`, `Release-vs-Nightly-Build`, `GUI-Validation-Methods`, `MoreData`, `Development-Notes`. Internal links use bare page names; code links point at `blob/master/...`. Wiki numbers can lag the repo by a few commits ("approximately" is acceptable); repo PR numbers must match the PR's base exactly. Re-verify every count against `origin/master` first.

### GUI validation & screenshots

- **Headless Avalonia screenshot via `--screenshot-all` is the most reliable path** and works even on a locked desktop (it uses Avalonia's own off-screen `RenderTargetBitmap`, not the screen). Run the (worktree's) built exe: `FEBuilderGBA.Avalonia.exe --rom <ABS path> --screenshot-all --screenshot-dir=<ABS tmp> --screenshot-tab=<TabAutomationId>`. It calls `SelectFirstItem()` so captures have populated data; output is `<dir>/Avalonia_<ViewClassName>_<romVersion>.png` (the View name is the `GetAllEditorFactories()` registration string). It captures ALL ~356 editors (slow, ~6–8 min, saves incrementally) — run in background and poll for your editor's file by name, then `pkill`. Other flags: `--screenshot-show-panel=<AutomationId>`, `--screenshot-invoke-button=<AutomationId>`. Use `git add` of explicit files (the screenshot dir is untracked but not gitignored). The exe must be the build that includes your change, with an absolute `--rom` path. The two `[AvaloniaFact]` render paths (`RenderTargetBitmap.Save` and `view.CaptureRenderedFrame()`) NO-OP locally because the test harness uses `UseHeadlessDrawing` — the data-population asserts are the real proof; don't treat a missing PNG as a code failure.

- **No-ROM screenshots** (`roms/` empty, editor is a `Window`): write an `[AvaloniaFact]` test that builds a synthetic in-memory ROM (`new ROM()` + `SwapNewROMDataDirect(byte[])`) and populates the Core state the editor reads. The default test harness renders blank — temporarily flip `FEBuilderGBA.Avalonia.Tests/TestApp.axaml.cs` `BuildAvaloniaApp()` to `.UseSkia().UseHeadless(new AvaloniaHeadlessPlatformOptions { UseHeadlessDrawing = false })`, run the one screenshot test with `FEBUILDERGBA_SCREENSHOT_DIR=<abs>`, then REVERT `TestApp`. A `Window`-derived view renders blank if rendered directly — mirror `App.RenderMainViewToPng`: `view.Position = new PixelPoint(-4000,-4000); view.Show(); view.UpdateLayout();` before rendering, `view.Close()` after.

- **Locked-session GUI validation.** A locked Windows session makes MCP screen capture return all-black (the secure desktop) and MCP mouse/keyboard/`focus_window` dead no-ops; `get_screen_size` returns 640x480. What still works locked: `tools/WinCapture` (PrintWindow API — `PrintWindow(hwnd, hdc, PW_RENDERFULLCONTENT)` renders the window into a DC regardless of screen lock; build via `dotnet new console` + `System.Drawing.Common`, then `dotnet run --project tools/WinCapture -c Release -- "<window title>" out.png`), and PowerShell UIAutomation `InvokePattern.Invoke()`/`SelectionItemPattern.Select()`/`ValuePattern` (the accessibility layer, not display). Modal dialogs opened via `Window.ShowDialog` (e.g. `NumberInputDialog` count-prompts in List-Expand flows) do NOT render or activate on a locked session — don't burn time driving them; rely on unit/round-trip tests as the authoritative proof and capture the affected editor itself with an honest note. Click-driven jumps wired to a `TextBlock`/hyperlink are un-triggerable locked (a TextBlock has no `InvokePattern`).

- **PowerShell UIAutomation is the primary GUI validation method** (PR #249 added 3,132+ AutomationIds). Open any editor directly via its MainWindow button (`Main_<Editor>_Button` AutomationIds) + `InvokePattern.Invoke()` — no filter typing. Most editors auto-select row 0 on `Opened`→`LoadList`, so the editor opens populated; do NOT fight UIAutomation to select a list row (`SelectionItemPattern.Select()` on an Avalonia ListBox does NOT raise `SelectionChanged`, so `LoadEntry` never fires — drive selection via the prev/next nav buttons whose `Click` handlers set `SelectedIndex` in C#). `ValuePattern.SetValue` on a SearchBox triggers the live filter. NumericUpDown value isn't exposed via `ValuePattern` (composite control) — read the inner TextBox or the ViewModel. ComboBox dropdown items are virtualized (unreliable to scroll). Stall-signal caution: a 0-byte agent transcript + quiet file-mtime are UNRELIABLE stall signals (three agents `TaskStop`'d on those signals were all alive and near-done) — only stop on clear extended silence (~15+ min no new git-modified files AND no process AND no transcript growth).

- **Screenshot honesty.** Never fabricate screenshots; if locked/un-capturable, say so and point to the round-trip tests. Patch-gated editors (CSkillSys/SkillSystem) render their banner + empty list on commercial (unpatched) FE8U/FE8J ROMs — that still proves the structured layout replaced an inert placeholder. For populated data, drive a synthetic patched ROM through the VM in a test and use the test log (e.g. `listCount=255`) as the data-population proof. The screenshot test regenerates the PNG every run; add it to `.git/worktrees/<wt>/info/exclude` to stop `git add -A` churn. A GUI fix needs GUI-validated proof, not just a Core/CLI test (the bug often spans parse AND data-load). When a bug report has screenshots, download and VIEW the images and verify against a real render before fixing — text/WebFetch can't see images, and "fixing" from text alone mis-targets.

### Decomp project

- **`.mar` map-layout round-trip honest framing (#1148 / PR #1346).** Format: the decompressed ROM blob is `[width:u8][height:u8]` then `w*h` raw u16 LE entries at offset 2; the `.mar` body is `(ushort)(rawTile << 3)` per entry, with dims carried in a sidecar `.mar.json` (`{width,height,srcAddr,format:"febuilder-mar-u16-shl3"}`). Claimable: the `.mar` u16 body round-trips byte-identically (`>>3`/`<<3`) and the raw uncompressed `[w][h]+entries` blob is byte-reconstructable. NOT claimable: no byte-identical compressed/ROM round-trip (FEBuilder's greedy LZ77 packer + `%4` padding ≠ the original ROM packer — write no compressed `.tmap.bin`); and because `<<3` truncates bits 13-15 at export, `ExportMap` must REJECT tile index `>= 0x2000` rather than emit a silently-lossy `.mar`. `ImportMap` has no ROM param, never touches `CoreState.ROM`, never throws; `--out` is project-contained via `DecompProjectDetector`/`ResolveSourcePath` (rejects abs/`..`); `--in` is intentionally not containment-checked. Register `--import-asset`/`--roundtrip-asset` BEFORE the bare `--project` rom-info fallthrough (same swallow hazard as `--export-asset`). The maintainer reopens over-claims — source-back only where a SAFE owner exists, keep the rest explicit export-only with tests proving no silent preview-ROM mutation.

- **Shops have no in-place C-array source owner (#1149).** Shop inventories are variable-length, sentinel-terminated (0x00) 2-byte (itemId+quantity) lists reached only via 3 scattered pointer paths (hensei `item_shop_hensei_pointer`; FE8 worldmap-point shop pointers at +12/+16/+20 in 32-byte entries; per-map event-cond OBJECT records at +4 of 12-byte records). The `fireemblem8u` decomp tree declares no `gShopData[]` C array, so the `DecompSourceWriterCore` in-place row rewriter is not applicable. Honest path = an EXPORT/migration artifact (an EA `.event`-style installable recreating each shop's list at its pointer slot), mirroring `DecompAssetExportCore` + `--export-asset`. Keep the in-place writer out of scope and the #1159 ROM-mutation guard.

- **Symbolic FE8U shop-list writer (#1354).** Canonical `fireemblem8u/master` shapes: `include/constants/items.h` is an `enum { ITEM_NONE = 0x00, ... }` (C auto-increment for bare members — NOT `#define`, so `.map`/`.elf` symbol resolvers don't cover it); `src/worldmap_shop_data.c` is item-id-only `CONST_DATA u16 ItemList_X[] = { ITEM_SWORD_IRON, ITEM_NONE, };` (no quantity). Implemented as `DecompConstantResolver` (read-only, never-throws, per-project cache). Key rules: parse the enum form, not just `#define`; symbolic source is id-only, so reject any nonzero high byte (quantity) with an actionable message (literal path keeps quantity); SCOPE the resolver to the `ITEM_` identifier prefix (a naive parser maps sibling constants like `FALSE=0`/`NULL=0` and emits the wrong `ITEM_NONE` terminator) — a sibling enum member still advances the auto-increment counter but is never mapped; an ambiguous (non-literal) member value poisons the running counter until the next explicit literal re-anchors. A tolerant C-constants parser must be comment/string-aware at EVERY scan site (the enum-body brace search must skip `{` inside comments/strings via `SkipTriviaAndLiterals`/`FindOpeningBrace`), and reverse id→name display must prefer the canonical `ITEM_NONE` over declaration order. Discovery precedence: owner `constantsHeader` → manifest `artifacts.itemConstants` → default `include/constants/items.h`; an explicit-but-bad path ⇒ resolver Unavailable, no fall-through to the default.

- **Portrait composite-sheet geometry (#1350).** The canonical FE7/8 portrait sheet is 128x112 indexed; authoritative slot coords live in `PortraitImportPreviewCore.cs` (:60-92) and `PortraitRendererCore.SplitPortraitSheet` (:588-602) — cite, don't invent: base face (0,0) 96x80; mini/chibi (96,16) 32x32; half-closed eyes (96,48) 32x16; closed eyes (96,64) 32x16 (FE7/8 only — FE6 omits eye slots, `IsFE6Image`); mouths 1..7 at (0,80)(32,80)(64,80)(96,80)(0,96)(32,96)(64,96) each 32x16. Frame index meaning is in `PortraitFrameStrings.cs`. Trap: `--export-portrait-all` writes ONE composited mug per slot, NOT a split multi-file package — there is no shipped "portrait package" file-set, and a single sheet with no per-frame manifest cannot verify frame ORDER, only slot bounds/dims/bit-depth/palette (Copilot rejects "misordered frame" claims). `--validate-asset --kind=portrait-package` is read-only, no ROM; match the sidecar `.pal` by sheet NAME (`Path.ChangeExtension(sheetPath, ".pal")`), not the first-sorted `*.pal`.
- **Avalonia portrait import color-key + 96x80 round-trip (#1750/#1751).** The standard FE7/8 Avalonia portrait importer accepts 128x112 FEditor sheets and 96x80 composed face PNGs. Before quantizing/remapping, the importer treats the first opaque WinForms-style background corner (top-right, then bottom-right, then top-left) as the transparent color key and maps exact RGB matches to palette index 0. A 96x80 face is copied to the 128x112 sheet canvas at (0,0), then reverse-assembled into the 256x32 D0 sprite sheet; D4/D12 are cleared because a face-only export has no mini/mouth-frame art to restore. Unsupported portrait sizes are rejected instead of linear-encoded into D0.
- **Avalonia FE-Repo Hackbox portrait import (#1752).** The shared Avalonia portrait importer also accepts FE8 HALFBODY 160x160 Halfbody Hackboxes when the HALFBODY portrait-extension patch is installed. It mirrors WinForms `BuildPortraitSeetHalfBody`: the D0 sheet is raw `{00 04 20 00}` + 256x64 px tiles, D8 is a 64-byte two-bank palette (rows 0-31 bank 0, rows 32-63 bank 1), D4 is compressed mini, and D12 is raw mouth frames. 144x304 Twoparter Hackboxes are intentionally rejected until a verified PART1/PART2 layout exists.
- **Portrait Import Wizard shared preview cache + cyclic frame nav (#1980/#1981/#1985).** The wizard's Step-2 source preview and per-frame live preview (`PortraitImportPreviewCore.RenderFramePreview`) previously diverged: `RefreshFramePreview` read the raw, un-keyed `_vm.LoadedImage.IndexedPixels`/`GBAPalette` directly, while the Step-2 preview went through `PortraitImportHelper.BuildPreviewImage`'s color-key + auto-quantize pipeline first — so an opaque non-zero background rendered transparent in one pane and opaque in the other. Fix: `PortraitImportHelper.BuildPreparedPreviewLoadResult(loadResult)` is the single color-key+quantize step (never mutates its input; returns a fresh `LoadResult`, or `null` on unusable input), and `BuildPreviewImage` now delegates to it + `BuildPreviewImageFromPrepared`. `ImagePortraitImporterView` caches ONE prepared result per load in `_preparedPreview`, rebuilt via `RebuildPreparedPreview()` exactly once right after each `_vm.LoadedImage` assignment (never per frame/crop NUD tweak), and cleared to `null` automatically on unusable input via the helper's own null-return contract. `RefreshFramePreview` and the Step-2 preview both read `_preparedPreview` — `ImportPortrait` still receives the original, un-prepared `_vm.LoadedImage` so ROM-write behavior is unchanged (the prepared cache is preview-only and must never reach the write path). Separately, the Frame NUD (`FrameInput`) is now `FEBuilderGBA.Avalonia.Controls.WrappingNumericUpDown` (`StyleKeyOverride => typeof(NumericUpDown)`, so it keeps the stock NumericUpDown theme): increasing past the max frame (10) wraps to 0 and decreasing past 0 wraps to 10, instead of clamping/stopping. Avalonia 11.2.3's `NumericUpDown.Spinner`/`SetValidSpinDirection()` are private, so the control reacquires `PART_Spinner` itself in `OnApplyTemplate` and re-widens `Spinner.ValidSpinDirection` after every base `On*Changed` callback (transient-free — no dispatcher hop, so the RepeatButtons never visibly flash disabled at a boundary). ### Core seams & ports

- **ProcessRunner POSIX worker containment (#1993).** `process_worker.py` calls `setsid()` before `exec`, so its PGID equals its PID. If the short-lived leader exits before `PosixProcessGroupContainment` observes that group, synchronous cleanup performs exactly one direct bounded `kill(-pgid, SIGKILL)` attempt: success/`ESRCH` becomes idempotent success, bounded `EINTR` retries, and `EPERM`/other errors fail closed. If activation is unobserved while the leader is still alive, the fallback state is consumed as `Failed` before returning so timeout/output cleanup and the retained reaper can never signal that PGID later. The state gate also prevents `finally`/`Dispose` re-signals; a failed normal-exit containment result is merged into `TerminationFailed` and retained-reaper/error reporting rather than discarded. No signal-0 probe is used because it would add probe-to-signal TOCTOU; the residual leader-reap-to-direct-signal PGID reuse window is inherent to numeric POSIX group signaling and is minimized by the one synchronous attempt.

- **Random-map FEMapCreator seam (#1978 Phase 1).** The Core adapter is a strict external-process boundary — FEBuilderGBA must NEVER vendor/download FEMapCreator code or assets, and it must NEVER shell out through `cmd`/`sh -c`. - **Ephemeral port exhaustion.** `gh`/`git` HTTPS calls to GitHub intermittently fail with `connectex: Only one usage of each socket address ... is normally permitted` (WSAEADDRINUSE) — an external daemon holds tens of thousands of ESTABLISHED connections, exhausting the dynamic port range. It is NOT TIME_WAIT (waiting doesn't clear it) and not caused by your own processes. Treat it as transient: wrap critical one-shot calls (`gh pr merge`, `gh api .../merge`, `git push`/`git fetch`) in a retry loop (~20 attempts, ~8s apart, break on success); prefer fewer, spaced-out calls; avoid tight polling. Diagnose with `Get-NetTCPConnection | Group-Object State`.

- **Near-full-disk survival** (concurrent worktrees can tip the machine to <500 MB free): agents build in their own worktree and CI is the authoritative build — a full local disk degrades but doesn't block (agents push and rely on CI). `bin/obj` file handles are held by lingering Roslyn/MSBuild build servers ("Device or resource busy" on rm) — `dotnet build-server shutdown` releases them. Only reclaim a worktree AFTER that agent's final completion notification (force-removing a still-running agent's worktree tears it down mid-task). Cleanup cadence after each merge: `build-server shutdown` → `git worktree remove --force` → `git worktree prune`. Keep waves to ~4-6 concurrent.

- **The environment wraps Claude Code in a corporate launcher** that transiently injects daemon hooks into `settings.json` at launch and removes them at exit — such hooks appear in NO static config file yet fire live (don't grep settings files for them). A known harmless error: a daemon hook runs a `DefenderAgentScan.exe` path containing a space whose quotes are lost when wrapped for the shell → a non-blocking "is not recognized" error (cosmetic). The toggle `AGENCY_SKIP_CLAUDE_SETTINGS=1` skips the whole settings-injection step, but it also disables all managed settings/telemetry/policy — clear with the relevant team before persisting.

## Working Conventions & Process Lessons

This section captures hard-won process and tooling conventions learned over the life of the project. The repository's `CLAUDE.md` already documents the baseline rules (committing under the project maintainer identity, one isolated git worktree per task, targeting the fork for all `gh` operations, the pre-commit checklist of three test projects plus screenshots, and always following `DEVELOPMENT-WORKFLOW.md`). What follows is the *nuance* behind those rules and the additional lessons that are easy to get wrong.

### Verification & Proof of Correctness

- **Never close an issue or declare work "done" without command output that proves each acceptance criterion.** "I read the file and it looks right" is not verification; "I ran `git check-ignore` on 31 paths and all returned ignored" is. List every acceptance criterion explicitly, run a verification command for each, and paste the evidence into the closing comment. If a criterion cannot be verified, do not close — state what is missing.

- **A GUI fix needs GUI-validated proof, not just a passing Core/CLI test.** A green unit test on an extracted Core helper is necessary but not sufficient — the editor can still be broken end-to-end. A real example: a Patch Manager `[HardCoding]` filter fix (PR #1384) made the token parser correct with green Core + parity tests, but the editor still showed "(filtered: 0)" because a *separate* layer (`PatchMetadataCore.EnumeratePatches`) only scanned immediate subdirectories and never loaded the deeper hardcoding patches; the real fix (PR #1386) was a recursive `SearchOption.AllDirectories` scan matching the WinForms `PatchForm.ScanPatchs`. Before merging a GUI fix, require proof the *specific editor produces the correct result* — a headless or MCP screenshot of the working editor, or an integration test exercising the full load → filter → display path. A unit-test-only proof on a GUI fix is a yellow flag: the bug often spans two layers (parse *and* data-load).

- **Unit tests verify code logic; functional tests verify the user experience — both are required for GUI work.** A zoom fix once shipped with seven passing unit tests and reviewer approval but a still-broken feature, because `Stretch="None"` meant the image never actually scaled and the app was never launched. Static offset audits (comparing WinForms vs Avalonia struct offsets) catch structural bugs but miss runtime/rendering bugs. For every Avalonia GUI change, launch the app with a real ROM and visually confirm the function works; a screenshot is the test, not an optional extra.

### Screenshots

- **When a bug report includes screenshots, download and actually VIEW each image before designing a fix, then verify the fix against a real rendered screenshot.** Text-extraction tools (e.g. WebFetch) cannot see images — never treat their text summary as having "seen" the screenshots. In one macOS Avalonia campaign, twelve issues were filed and "fixed" from the report's text alone; several mis-targeted the wrong UI element (e.g. a Class Editor fix widened the *Description* column when the screenshot plainly showed clipped *labels* in a different panel). To apply: resolve album links to a direct image URL (for Imgur, `curl -sL <album> | grep -oE 'content="https://i\.imgur\.com/[^"]+"'` yields the `i.imgur.com/<id>.png` URL), `curl -o` it, and Read the file. Reproduce locally with a headless screenshot of the editor against a real ROM, compare to the reporter's image, and only claim the fix done after a side-by-side (reporter image vs. your rendered screenshot) shows it resolved — not on green tests.

- **`feat`/`fix` PRs must include a screenshot of the SPECIFIC affected editor with real data loaded** — never a generic main-window shot, and never a CLI-output substitute when the changed files live under `FEBuilderGBA.Avalonia/`. `docs`/`chore` PRs (documentation, `.gitignore`, CI config, dependency bumps) are exempt.

- **Never fabricate screenshots.** Generating synthetic terminal-output images (e.g. `System.Drawing.Bitmap` + `DrawString()` showing "tests passed") and passing them off as GUI proof is cheating — it proves nothing about the GUI. If the screen is locked or MCP is unavailable, say so honestly and mark manual verification as TODO. Headless tests are supplementary proof, never a substitute for visual verification. (Note: `PrintWindow`-based capture works even on a locked screen, so "screen locked" rarely justifies skipping a real capture.)

- **Screenshot URLs in PR bodies must point at the default branch, never a feature-branch `blob/` URL** — this failure recurred across many PRs despite docs, skill, and lint reminders. The CI validator (`scripts/validate-pr-screenshots.sh`) rejects `blob/{ref}/` and `raw.githubusercontent.com/{owner}/{repo}/{ref}/` forms for any non-default branch. Two durable options: (1) commit the image to `pr-screenshots/` via a separate docs PR, merge it, then reference `blob/master/pr-screenshots/...`; or (2) upload it as a release asset and reference it with an `<img src="https://github.com/<owner>/<repo>/releases/download/<tag>/<name>.png">` tag (release-download URLs and `<img src=` tags both pass the validator). CI gotcha: the validator reads the *frozen event-payload* PR body, so editing the body with `gh pr edit` does **not** re-trigger it and `gh run rerun --failed` reuses the stale payload — push a new commit (an `--allow-empty` chore commit works) to fire a fresh `synchronize` event.

### Diagnosing Blocked PRs & Review Threads

- **Never guess why a merge is blocked — diagnose systematically.** Past failures all came from assuming ("needs approval") instead of checking, when the real cause was an outdated branch or an unresolved bot comment. Use `gh pr view <N> --json mergeStateStatus,statusCheckRollup` for the actual block reason, and check causes in order: unresolved threads → CI status → branch up-to-date → review approvals. Also confirm which branch you are on before changing anything.

- **Check issue-level PR comments, not just inline review threads.** Human reviewers, the Copilot bot, and the Copilot CLI all post non-inline comments that the standard `reviewThreads … isResolved==false` check misses entirely. After every push run *both*: `gh api repos/<owner>/<repo>/issues/<N>/comments` (filtering out `github-actions[bot]`) for non-inline comments, and the GraphQL `reviewThreads` query for inline ones. Address all comments from all sources before re-review or merge.

- **Fix the code before resolving any review thread — never acknowledge-dismiss.** The correct order is fix → push → re-review → resolve, never "skip the fix and resolve." But the inverse mistake also happened: a thread-fix that was complete yet left every thread open ("let the bot re-resolve") kept a PR `BLOCKED`, because the bot does *not* auto-resolve after re-review. The author owns the resolution click: after the Copilot CLI re-review reports no blocking findings and CI is green, manually resolve each addressed thread via the `resolveReviewThread` GraphQL mutation. For a genuinely out-of-scope concern, file a follow-up issue, post one inline reply referencing it, then resolve. Confirm `mergeStateStatus: CLEAN` before merging.

- **Auto-merge `OPEN · MERGEABLE · CLEAN` PRs with green CI and no unresolved threads — do not stop at "ready to merge."** The workflow's stated goal is MERGED. Use the repo's squash convention (`gh pr merge <N> --squash`); recent merges all carry a `(#NNN)` suffix. Stop and ask only when CI is failing, threads are unresolved, `mergeStateStatus` is `BLOCKED`/`DIRTY`, or the change is high-risk (force-push, public API change).

### Scope & Tracking Deferred Work

- **Do exactly what was asked — do not expand scope.** When told to cancel "E2E CI runs," cancel only those; cancelling an adjacent required "Check" run blocked merges that were never in scope. Filter precisely by name/type; if unsure whether something is in scope, ask first.

- **Do not silently shrink scope either — file deferred work as GitHub tracking issues.** When a plan lists "out of scope" / deferred items, open one issue per item (clear title, 2–3 lines of context, link the originating PR/plan and why it was deferred) and reference the issue numbers back in the PR/plan. Deferred ideas are real backlog; leaving them in a plan file or PR comment loses them.

- **When implementing a previously-deferred feature, grep ALL layers for the stale "deferred" claim — not just the files you edit.** The same "intentionally deferred" doc-comment is typically duplicated across the Core helper, the `*View.axaml` markup header, the `*View.axaml.cs` code-behind, and the `*ViewModel.cs`. A GoldRoad implementation (#1244) removed the claim from three files but missed the duplicate in the `.axaml` markup. Before committing a "now implemented" change, run a repo-wide grep for the stale phrase across `--include=*.cs --include=*.axaml` and verify zero remaining "deferred"/"follow-up" hits; the `.axaml` markup files are the easiest to forget.

### Worktree Isolation, Commit Timing & Recovery

- **`isolation: "worktree"` can silently fail and land work in the main checkout.** A real worktree's spawn result shows `worktreePath` under `.claude/worktrees/agent-*`; when that line is *absent* the agent may have run `git checkout -b` in the shared main checkout, switching its branch and breaking other sessions. After every worktree-agent run, confirm the `worktreePath` line; if absent (or the agent reports "main checkout"), run `git -C <main> branch --show-current` and `git worktree list`, and if the main worktree was switched, restore it with `git checkout master && git pull --ff-only`. Two extra failure modes: a *leader* session that is itself inside a worktree must give each spawned agent a **unique** branch name (the harness cannot create a second worktree for an already-checked-out branch); and a parallel Copilot CLI `--allow-all-tools` review's own worktree cleanup can wipe the files of an *active* implementation worktree — never keep un-pushed work alive while such a review runs.

- **`Write`/`Edit`/`Read` tool calls inside a worktree must use the WORKTREE absolute path, not the main-checkout root.** Most documented absolute paths point at the main checkout; copying them into the path tools silently writes files onto `master`, invisible to the worktree, and `dotnet test` in the worktree then reports "No test matches." (`Bash` heredocs with relative paths after `cd <worktree>` land correctly; only the path tools mis-route.) Derive every `file_path` by prefixing the worktree root.

- **Commit and push the instant the changed code's targeted gates pass — before any side quest.** A pushed branch is the unit of progress; an uncommitted worktree is invisible to a lead and blocks merge, and in a shared session the worktree can be *deleted out from under you* mid-task (directory gone, branch unregistered, no reflog) when parallel orchestration merges the same task. Run the targeted gates (the broad `~Avalonia|~AutomationId` filter, the feature's own Core/Avalonia tests, `validate-automation-ids` 0/0, a clean build), then commit and push immediately; screenshots, follow-up issues, and PR creation all come *after*. Do not block the push on a redundant full-suite rerun (CI reruns it anyway). If the worktree vanishes before committing but the file contents are still in context, recovery is cheap: first check `git -C <main> log` and `gh issue view <N> --json state` (the task may already be merged — don't recreate duplicates), then recreate a fresh worktree off `origin/master` and re-author.

- **Headless screenshots need a real ROM, which a worktree lacks.** `git worktree add` does not copy gitignored files, and the ROMs under `roms/*.gba` are gitignored, so `--screenshot-all --rom "<worktree>/roms/..."` exits with an error. Point `--rom` at the **main checkout's** ROM, or copy the ROMs into the worktree first. This is the concrete reason screenshot capture is a separate, fragile step that should run after the commit, not gate it.

### Test Discipline

- **Run all three desktop test projects before every push — not just the two cross-platform ones.** `FEBuilderGBA.Tests` (`net10.0-windows`) contains cross-project assertions about Avalonia code, including *source-text* assertions that grep specific files for literals. When a refactor relocates such code into a new Core seam (e.g. moving wait-icon decode literals into `WaitIconRenderCore.cs`), the source-text test fails *only* in `FEBuilderGBA.Tests`, which is part of CI's `build` job — so "green locally" from Core.Tests + Avalonia.Tests alone goes red in CI. Update the relocated source-text test to assert the same contract in the new seam, and always run:
  ```
  dotnet test FEBuilderGBA.Core.Tests/FEBuilderGBA.Core.Tests.csproj
  dotnet test FEBuilderGBA.Avalonia.Tests/FEBuilderGBA.Avalonia.Tests.csproj
  dotnet test FEBuilderGBA.Tests/FEBuilderGBA.Tests.csproj
  ```

- **Run EVERY test case in the plan and list each PASS/FAIL before opening a PR** — not a subset, not "the important ones." Code being written is not the finish line; validation evidence is.

### Avalonia / AXAML CI Traps

These are silent at compile time and only surface in CI or Copilot review:

- **`Log.Error`/`Log.Notify`/`Log.Debug` do NOT apply composite format strings.** They are `params string[]` joined with spaces (`FEBuilderGBA.Core/Log.cs`), so `Log.Error("X failed: {0}", ex.Message)` logs a literal `{0}` and drops the stack trace. Use one concatenated/interpolated string with `ex.ToString()`.

- **AXAML `StringFormat` is not translated.** The translation pass only translates `Text`/`Header`/`Content`/`Title`/`Watermark`/`ToolTip` *values*, so a `StringFormat='{}{0} change(s)'` leaves the literal English in ja/zh and fails `L10nCoverageTest`. Build the fully-localized string in the ViewModel via `R._("{0} change(s)", n)` and bind the property, and add the key to `config/translate/{ja,zh}.txt`.

- **Every `R._("English literal")` in `FEBuilderGBA.Avalonia/**/*.cs` needs ja+zh entries** or the cross-platform `build` job fails (`CodeLiteralL10nCoverageTest`). The scanner keys on the *exact* runtime string (including real newlines, em-dashes U+2014, and straight quotes), so keep each `R._` a single-line phrase and concatenate any `{0}`/variable/newline part *outside* `R._`. Wrapping a dialog in `R._` without adding the translation entries is the common miss.

- **Hardcoded hex `Background`/`Foreground` in a View AXAML fails CI** via `AvaloniaDarkModeTests.NoHardcodedHexColorsInViewAxamlFiles` (regex `(Background|Foreground)="#([0-9A-Fa-f]{6,8})"`; only 8-digit ARGB with low alpha `^[0-3]` is exempt). This runs *only* in the `net10.0-windows` E2E `build` job, so the cross-platform matrix, Android, and gap-sweep all pass while the PR stays blocked. Use a `{DynamicResource …Brush}` theme key and named brushes (`BorderBrush="Gray"`); named colors and `BorderBrush=` are not matched.

- **A pointer/hex field in an Avalonia editor must use a hex `TextBox`, not a `NumericUpDown` with a hex `FormatString`.** `NumericUpDown.Value` is `decimal?`, so `FormatString="X8"` throws `FormatException` at render — caught only by `AvaloniaEditorTests.NoAxamlFile_UsesHexFormatString_OnNumericUpDown`. Use a `TextBox` (watermark `0x00000000`, parse via a local `ParseHexText`, display via `"0x{0:X08}"`) and put any read-only address in a separate `TextBlock`. Include `AvaloniaEditor` in the pre-push test filter — the other Avalonia gate filters do not catch it.

- **Every `<Window>` view must carry a `SizeToContent=` attribute** (`AvaloniaScrollViewerTests.AllWindowViews_HaveSizeToContent`). For a window that must be fixed-size, set `SizeToContent="Manual"` plus a `MinWidth`/`MinHeight` floor — do **not** delete the attribute (deleting it and adding a test exemption is rejected, since the views still declare `SizeToContent`).

- **A read-only aggregator/tool ViewModel that reads no ROM bytes must NOT implement (or even mention in comments) `IDataVerifiable`.** `AvaloniaFieldCompletenessTests.NoOrphanVMs_ImplementIDataVerifiable` does a plain `source.Contains("IDataVerifiable")`, so even an XML-doc comment naming it trips the check for any VM not in `FormMappings` or factoried in `MainWindow.axaml.cs`. Such VMs are registered for navigation via `wm.Open<XxxView>()` (registration ≠ FormMappings); keep a plain `GetListCount()` if a test needs one.

A good pre-push habit: grep for `Log.\w+\([^)]*{0}`, `StringFormat=`, and `(Background|Foreground)="#"`, and run `dotnet test FEBuilderGBA.Tests/FEBuilderGBA.Tests.csproj --filter "FullyQualifiedName~DarkMode"` (plus the broad `~Avalonia|~AutomationId` filter) locally.

### Destructive-Operation Safety & Cleanup

- **Never run an unguarded bulk `rm -rf` over computed/looped paths.** A slash-normalization (`sed`) error once left a loop variable empty, defeating the main-checkout exclusion, and `rm -rf "$var"` began deleting the main checkout — including the gitignored `roms/` folder (custom test ROMs, permanently lost since `rm -rf` bypasses the Recycle Bin and git cannot restore ignored files). Prune worktrees **only** with `git worktree remove --force <path>` followed by `git worktree prune`; never hand-roll `rm -rf` over a path list. If a cleanup script errors on any line, stop and inspect rather than proceeding to a destructive action with a possibly-garbage variable; prefer `set -e` or an explicit allow-list of safe path prefixes.

- **Never run `git clean -x`/`-X`/`-fdx` in this repo.** The ROMs (`roms/*.gba`) are gitignored, so `git status` does not list them, but `git clean` with `-x`/`-X` *removes ignored files* and wipes them. If you must clean untracked files, use `git clean -fd` (untracked, not ignored) and exclude the ROMs: `git clean -fd -e roms -e '*.gba'`. Treat `roms/*.gba` as protected and irreplaceable — committed work is always recoverable from the remote, but gitignored local files are not.

- **Proactively run the full end-of-task cleanup — do not wait to be asked.** After any non-trivial task (especially batches that spawn worktrees or merge PRs), run in order: (1) prune agent worktrees (`git worktree list` → unlock + `git worktree remove --force` + `git worktree prune`); (2) delete merged local branches (`git branch -D`); (3) run `git status` and resolve every untracked/unstaged file — agent setup commands like `dotnet new console --force` have clobbered real source with a Hello-World stub, so treat any unexpected change as a potential regression and restore it; (4) `git fetch` + `git pull --ff-only` (investigate before any `reset --hard`); (5) `git submodule update --init --recursive`; (6) final verify that the working tree is clean and `HEAD == origin/master`. After merging a PR, auto-prune its worktree immediately — all tasks use isolated worktrees, so the main worktree is never on a feature branch and the next task picks up fresh master automatically.

- **When disk space is low, prune stale worktrees first** — each carries a full checkout plus `bin`/`obj`/publish artifacts and is usually the largest reclaimable target. `git worktree prune` does *not* reclaim the bulk: when `git worktree remove` runs while a DLL is locked, it drops the admin record but leaves the directory, invisible to `git worktree list`/`prune`. After the git-level prune, sweep `.claude/worktrees/` directly. A worktree's lock reason lives in `.git/worktrees/<id>/locked`; if the named PID is dead the lock is stale (`git worktree unlock` then remove `--force`). Beware the squash-merge trap: merged branches appear "unmerged" via `git cherry`/`--merged` because PRs squash to a new commit — verify with `gh pr view <N> --json state,mergedAt` and confirm the branch's last commit predates `mergedAt`.

### GitHub & Commit Hygiene

- **Append the Claude Code version and model footer to EVERY public GitHub output** — discussion replies, issue comments, PR descriptions and comments, and review replies — and include it in the prompt text of any sub-agent that may post to GitHub (do not rely on the agent to add it). This is a zero-tolerance, blocking rule; verify each comment ends with the footer before reporting success.

- **Use the real `claude --version` output for the commit-message version line** — never hardcode a version string. Run `claude --version` before committing.

- **Never write a personal/real name into any committed file content** — code, docs, comments, config, or skill files. When documenting identity rules, phrase them as "use the project maintainer identity, never any other," and grep staged files for the real name before committing to catch accidental inclusion.

- **Keep `.claude/skills/dev-flow/SKILL.md` and `DEVELOPMENT-WORKFLOW.md` in sync in the same commit.** They define the same workflow, and the skill file is what the harness actually reads at session start — divergence causes the wrong rules to be enforced. Never commit one without the other.

### SkiaSharp ↔ Avalonia Native Version Conflict — RESOLVED (#798)

Any code that calls managed SkiaSharp directly inside the Avalonia process (the app or `FEBuilderGBA.Avalonia.Tests`) must use the same SkiaSharp major/native version Avalonia bundles. Avalonia 11.2.3 bundles native libSkiaSharp 2.88 ("88.1"), but `FEBuilderGBA.SkiaSharp` had been pinned to managed SkiaSharp 3.116.1; since one process loads only one native libSkiaSharp, the first managed-3.116 call threw on Linux/macOS. Resolution: pin `FEBuilderGBA.SkiaSharp.csproj` to Avalonia's transitive SkiaSharp 2.88.x (Avalonia 11.x is not compatible with SkiaSharp 3.x).

- **Most reusable lesson:** Windows masks the conflict (native resolution loads the 3.116 dll), so a Windows-only local run passes while ubuntu + macOS CI fail — always check all three platforms for SkiaSharp changes. Exact-byte cross-architecture (x64↔arm64) image goldens are *not* achievable (`SKFontHinting.None` makes x64 Windows==Linux byte-identical, but arm64 macOS AA crosses the threshold ~1–2 pixels differently); use a pixel-index tolerance (decode bytes → count differing pixels → assert `<= ~6`) instead of exact-equality, while keeping the robust structural asserts.

### Durable Engineering Lessons

These lessons are extracted from a running development journal and apply to ongoing work in this repo.

#### Avalonia CI Contract Gates (run locally before pushing)

Any new Avalonia `<Window>` view or control trips repo-level CI gates that are easy to miss locally (especially when patch2-submodule failures mask them):

- **AutomationId suffix gate** (`scripts/validate-automation-ids.ps1`): every `AutomationId` must end with an approved suffix — `_Input`, `_Combo`, `_Button`, `_List`, `_Check`, `_Expander`, `_TabControl`, `_Tab`, `_Image`, `_Label`, `_Control`, `_Link`, `_TopBar`. Note `_Check`, not `_CheckBox`. The `build` job fails fast (~47s) on a violation.
- **ScrollViewer gate** (`AvaloniaScrollViewerTests`): every `<Window>` must include `SizeToContent=` and a `ScrollViewer` (or be added to the exempt list).
- **L10n coverage gate** (`L10nCoverageTest`): every English UI literal in a `Views/*.axaml` `Text`/`Content`/`Header`/`ToolTip`/`Watermark` attribute needs `ja` + `zh` translations. Add a three-line block (`:<English>` / `<translation>` / blank) to `config/translate/ja.txt` and `zh.txt` (and `en.txt` for consistency); files are UTF-8, committed as LF (a Windows checkout may show CRLF via autocrlf — match the file's existing endings), and live in the MAIN repo (only `config/patch2` is a submodule). Strings built in code (VM strings, code-behind `.Text =`, detail-panel labels) bypass the AXAML scan — wrap each in `R._(...)` and add entries; avoid generic keys like `yes`/`no` that collide.
- **Field-completeness / dialog-dimension gates** (`AvaloniaFieldCompletenessTests`, `AvaloniaDialogViewTests`, `AvaloniaEditorTests`, all `net10.0-windows`): VM fields must match the WinForms Designer, and window dimensions must match the WinForms form. Running `AvaloniaFieldCompletenessTests` rewrites `docs/field-completeness-report.txt` with CRLF noise — `git checkout --` it before committing.
- **Cross-project source-text parity assertions** live in `FEBuilderGBA.Tests/Unit/Avalonia*Tests.cs` (a `net10.0-windows` project not in the quick Core/Avalonia pass). Removing a control `Name=` that a parity test pins turns CI red.

#### Logging and Formatting Traps

- `Log.Error` is `params string[]` — it joins arguments with spaces and does **no** `string.Format` substitution. Never write `Log.Error("...{0}", x)` (it logs the literal `{0}`); use `Log.Error("... " + ex.ToString())`. For user-facing C# strings use `R._(fmt, args)`, which both formats and localizes.

#### ROM Data-Model Parity (porting WinForms editors to Avalonia)

- An Avalonia editor VM that models a pointer-dereferenced ROM struct as inline fields will clobber the pointer and ASM handlers on Write (silent corruption). Always diff the VM against the WinForms form byte-offset by byte-offset. Field-name conventions: `B`/`W`/`D`/`P{offset}` = `u8`/`u16`/`u32`/`p32`. A `DWORD` field is a RAW `u32`, not a pointer-stripped `p32`.
- `ROMFEINFO` `*_pointer` fields are **pointer slots, not table bases** — always dereference first (`DerefPointer(rom, rom.RomInfo.X_pointer)` / `rom.p32(...)`) and bound by the matching `maxcount`. Reading the pointer field as a base yields garbage labels.
- The standard table-access idiom: `base = rom.p32(unit_pointer)`, then `base + i * unit_datasize`, bounded by `unit_maxcount`.
- For NewAlloc parity, WinForms `AppendBinaryData(new byte[size])` zero-fills first — port that (`rom.write_range(addr, new byte[size])`); a `0xFF` free region leaves handler slots as `0xFFFFFFFF`. Validate **all** command slots before writing the header, or a refused Write partially mutates the ROM.
- Undo coverage for class-C/D editors: copy the canonical sibling pattern verbatim — `UndoService.Begin(...)` / try-`Commit()` / catch-`Rollback()`, an `IsLoading`/`IsLoaded` guard, and post-commit `MarkClean()`. Canonical sibling views: `Command85PointerView`, `ClassEditorView`, `AIPerformItemView`, `EDFE6View`.
- A GUI fix usually spans two layers — VM logic **and** AXAML bindings. When a VM has a "warn the user" requirement, grep the `.axaml` for `Text="{Binding}"`; a view with no bindings leaves typed values unreachable.

#### Faithful File Export / Round-Trip

- When exporting ROM data to a file, read the original bytes (`getBinaryData`, with a length walker such as `RebuildProducerCore.CalcPathDataLength`) rather than re-serializing the edit model. Re-serializing diverges from WinForms semantics. Check Core for an existing walker before writing one.
- When switching a list loader (e.g., to expose all frames), grep for **every** surface that duplicates the buggy logic (`ListParityHelper.Build*`) and verify helper parameter **order**, not just the name — slot-order mismatches (image/TSA/palette, raw vs LZ77) are a frequent trap.

#### Cross-Platform Test Traps

- `U.escape_filename` and anything touching `Path.GetInvalidFileNameChars()` / `GetInvalidPathChars()` is OS-dependent (e.g., `*` is illegal only on Windows). For cross-platform bad-filename tests use NUL (`'\0'`), invalid on every OS, so the test does not pass locally on Windows but fail on macOS/Linux CI.
- Platform detection: use `RuntimeInformation.ProcessArchitecture` (running process bitness), not `OSArchitecture` — an x86 process on 64-bit Windows reports `OSArch=X64`.
- Tests that instantiate Avalonia Views (not just VMs) access `CoreState.ROM`; add ROM skip guards or they crash in CI without ROMs.
- Synthetic-ROM pointer slots must clear the `0x200` safety floor (`isSafetyOffset`), or every guard rejects them — real ROMs root these far higher. Use injectable filesystem abstractions (interface + real impl + in-memory fake) to make disk-mutating Core logic unit-testable.

#### Screenshots and the PR Screenshot CI Gate

- The screenshot validator (`scripts/validate-pr-screenshots.sh`) rejects any `raw.githubusercontent.com/{owner}/{repo}/{ref}/...` whose `ref` is not the literal default branch — feature branches, the `issue-assets` branch, and even commit-SHA-pinned URLs are all rejected. PR-body screenshot URLs must use `raw.githubusercontent.com/.../master/pr-screenshots/...` (a file added by the same PR resolves on merge; the check is format-only, not reachability). CHECK 2 (require ≥1 rendered image on feat/fix PRs) keys on the title regex `^(feat|fix):`, so a *scoped* title like `feat(avalonia):` is not matched by it. This is a known regex gap, not a sanctioned way to skip proof — feat/fix PRs should still include a screenshot per the contribution guidance. An alternative stable host is a GitHub release asset (`gh release upload <release> <png> --clobber`) embedded via `<img src=>`.
- GitHub re-runs replay the **original** event payload, so editing the PR body and re-running the same run keeps validating the stale body. To re-trigger CI on a body fix, push a new commit (a `synchronize` event); the `pull_request` workflow fires on `synchronize`/`opened`/`reopened`, **not** on `edited`.
- Master branch protection forbids direct pushes (including screenshots) — the reliable path is a release asset or a screenshot PR, never a direct push to `master`.
- **Screenshot privacy:** before committing a `--screenshot-all` PNG, view it and confirm it shows no `C:\Users\<name>\...` real path or username (file-field auto-fill leaks it). Fix the feature to not surface the loaded ROM's own path, then regenerate.
- **Headless screenshot mechanics:** headless Avalonia uses `UseHeadlessDrawing` (no Skia rasterizer), so `RenderTargetBitmap.Save` is a no-op and in-test screenshots are blank — that proves layout only, not data. For populated-data proof, drive the live app via UIAutomation (`ValuePattern.SetValue` + `InvokePattern`) plus `PrintWindow`/`WinCapture`, or render a SkiaSharp proof image from real ROM values through the production VM in a `[Fact]` (the test project has a transitive SkiaSharp reference). For a ROM-independent view, the `--screenshot-window=<View> --screenshot-out=` hook renders a real PNG without driving the main window. `--screenshot-all` hangs if preceded by a `--help` probe (which never returns) — run it directly. Render order follows menu order, not alphabetical. A locked screen produces pure-black MCP screenshots; `PrintWindow` + UIAutomation work while locked. `--screenshot-all` realizes a small window height that clips below-fold controls — scroll via UIA `ScrollPattern.SetScrollPercent(...,100)` before capturing.

#### Conventional-Commit and Subagent Workflow

- The commitlint type-enum gate allows only `build`, `chore`, `ci`, `docs`, `feat`, `fix`, `perf`, `refactor`, `revert`, `style`, `test`. `i18n` is **not** allowed. To fix a bad type non-interactively, reword via `GIT_SEQUENCE_EDITOR` (mark `reword`) + `GIT_EDITOR` (sed the type), then `git push --force-with-lease`; verify the tree is unchanged with `git diff old HEAD --stat` (empty = safe).
- Subagent result text can be lost (background-agent completion notifications fire but the transcript stays empty), while the agent's commits land fine. Do not trust notification timing — `git fetch` + `git log` the branch to confirm what actually committed/pushed (an agent whose commit "didn't show" had often already pushed under a different SHA).
- Worktree isolation can fail: a session pinned to one worktree cannot Edit/Write paths in another. Workarounds are to drive edits via Bash + Python exact-string replacement (`io.open(..., newline='\n')`, raw Python strings for literal `\n` in C# string literals) or to delegate to a worktree-scoped agent. When manually creating worktrees, never reuse the leader's branch name for a spawned agent, and always use `git -C <dir>` / absolute paths since the shell cwd resets between calls.
- Two recurring **false** Copilot findings to refute with a green build rather than churn: "uint won't index `byte[]`" (C# `byte[]` has a `uint` indexer) and similar cast complaints.
- GitHub issue-form URL prefill is real: `issues/new?template=X.yml&{field_id}={value}` works, where `field_id` is the canonical field identifier and dropdown values must match the option strings exactly.

#### Translation Pipeline

- The WinForms translation dictionary is keyed by **Japanese** strings; Avalonia uses English keys via a reverse English-to-Japanese map plus Avalonia-specific entries. A full i18n fix has three layers: (1) the reverse map for existing WinForms translations, (2) Avalonia-specific entries in the translation files, and (3) wiring AXAML hardcoded strings to `R._()` in code-behind. Trace the full lookup chain (`R._()` → `str()` → `Dic.TryGetValue()`) — "the file is loaded" does not mean "the key is found." Always visually diff before/after screenshots; "no error" is not "feature works."

#### Gap-Audit Hygiene

- Auto-generated gap-sweep reports go stale fast and are dominated by false positives (Japanese-WF vs English-Avalonia labels, differing undo patterns, scanner mis-bracketing). Prefer grepping developer-authored markers (`not.?implemented`, `deferred`, `KnownGap`, `use the winforms`, `stub`) and **verify the WinForms side actually has the feature** before treating an Avalonia stub as a gap — an Avalonia-only control with no WinForms counterpart is new-feature scope creep, not a gap. Before trusting any audit, confirm the base is fresh (`git rev-list --count HEAD..origin/master`). Before writing `Closes #N`, re-read the issue title word by word ("all forms" means all View classes, not all ViewModel classes) and verify claimed gaps with an actual diff, not stale audit numbers.
